## КОРНЕВАЯ ПРИЧИНА: два бага, которые складываются

### Баг №1: `read=120s` таймаут — СОКЕТ-уровневый, не потоковый

Это **главное открытие**. Я проследил полный путь таймаута от httpx до httpcore:

```
httpx.Timeout(read=120)
  → request.extensions["timeout"]["read"] = 120.0
    → httpcore AsyncHTTP2Connection._read_incoming_data()
      → self._network_stream.read(max_bytes, timeout=120)
        → AnyIOStream.read()  [ httpcore/_backends/anyio.py:33]
            with anyio.fail_after(120):   ← ТОЛЬКО ЗДЕСЬ
                return await self._stream.receive()
```

`anyio.fail_after(120)` оборачивает **один сокетный `receive()`** — одно чтение из TCP-сокета. Это **не** таймаут для конкретного H2-стрима. Это таймаут для всего сокета.

**Что происходит при 14 активных стримах:**

1. Стримы 1-8 получают данные от сервера (сервер обрабатывает их)
2. Стримы 9-14 ждут свои заголовки ответа (сервер поставил их в очередь)
3. Сокет **НЕ молчит** — для стримов 1-8 постоянно приходят DATA-фреймы
4. Каждый `self._stream.receive()` возвращает данные за миллисекунды
5. `anyio.fail_after(120)` создаётся заново на каждой итерации цикла
6. **Дедлайн 120 секунд НИКОГДА не достигается**, потому что сокет постоянно активен

Цикл ожидания в `_receive_stream_event` (` httpcore/_async/http2.py:335-337`):
```python
while not self._events.get(stream_id):    # очередь стрима 9 пуста
    await self._receive_events(request, stream_id)  # читаем сокет
    # сокет вернул данные для стримов 1-8 → events распределены по их очередям
    # очередь стрима 9 всё ещё пуста → цикл повторяется
    # НОВЫЙ anyio.fail_after(120) — таймер сброшен!
```

**Результат:** стримы 9-14 ждут **бесконечно**. Единственный таймаут, который срабатывает — это `asyncio.timeout(600)` в buffered-retry режиме ( `gateway_service.py:598`), который убивает весь retry-цикл целиком.

Это идеально объясняет твоё наблюдение: «стрим даже не начинается, код 200 не приходит, прокси не регистрирует ни байта» — для конкретного стрима никаких байтов нет, но **на сокете** байты идут (для других стримов).

### Баг №2: Пул доверяет advertised `max_concurrent_streams=100`

Логика принятия решений в пуле (`pool.py:155-165`):
```python
available_connection = next(
    (conn for conn in self._connections
     if conn.can_handle_request(origin)
     and conn.is_available()
     and connection_request_count[conn]
     < self._max_concurrent_requests(conn)),   # ← 100
    None,
)
```

`_max_concurrent_requests(conn)` → `FixedHTTP2Connection.max_concurrent_requests()` → `self._max_streams` = `min(server_advertised, 100)` = **100**.

Пул **никогда не откроет новое соединение**, пока на существующем не будет 100 активных стримов. С 10-14 агентами это никогда не происходит — все запросы падают на одно H2-соединение.

---

## Нет хардкодов в нашем коде, НО...

Я подтвердил: **число 100 не захардкожено в нашем коде**. Оно приходит из двух мест в библиотеках:

1. **h2** ( `h2/connection.py:356`): `MAX_CONCURRENT_STREAMS: 100` в `H2Connection.__init__`
2. **httpcore** ( `httpcore/_async/http2.py:204`): переустанавливает `MAX_CONCURRENT_STREAMS: 100` в `_send_connection_init`

Наш `FixedHTTP2Connection` **НЕ переопределяет** `_send_connection_init`. Значение 100 — библиотечный дефолт.

**В конфигурации нет настройки для `max_concurrent_streams`:**
- `HttpClientPoolConfig` (`schemas.py:462-475`): только `max_connections=100`, `max_keepalive_connections=20`, `keepalive_expiry=5.0`
- `HttpClientConfig` (`schemas.py:478-492`): только `http2: bool=True`, `pool`, `pool_health_log_interval_sec`
- Флаг `http2` — **глобальный**, не per-провайдер. Нельзя отключить H2 только для qwen-home

## Почему другие провайдеры работают

Другие провайдеры работают без проблем, потому что:
- Либо у них меньше трафика (меньше параллельных запросов, ниже реального лимита сервера)
- Либо их upstream API реально поддерживает 100+ параллельных стримов
- Либо у них меньше time-to-first-byte (сервер быстро отвечает, стримы не зависают)

---

## Почему перезагрузка помогает

Рестарт убивает все существующие H2-соединения (на которых висят "голодные" стримы в event loop). Новые соединения создаются чистыми, без зависших стримов. Цикл начинается заново — работает, пока не превысит ~8 параллельных запросов на одно соединение.

---

## Почему 4 воркера не спасают

Каждый uvicorn worker — отдельный процесс со своим пулом (` main.py:21-24` — `config`, `accessor`, `app` создаются на уровне модуля, форкаются 4 раза). Но:

- 10 агентов распределяются неравномерно между 4 воркерами
- Один воркер может получить 8+ запросов одновременно
- Лог показывает `streams: 14 active` — это один воркер с 14 стримами на одном соединении
- При лимите сервера ~8, 6 стримов зависают

`queued: 0` в логе — это не баг. Запросы не в очереди пула — они уже назначлены на соединение (стримы открыты), но сервер не отвечает на них. Пул видит их как "active", а не "queued".

---

## Твоя гипотеза про пул = 5: ДА, это сработает

Если бы мы могли ограничить `_max_streams` до 5 (или любого значения ≤8):

1. После 5 стримов на одном соединении пул открыл бы **новое TCP/H2-соединение** (`pool.py:170-174`: `elif len(self._connections) < self._max_connections`)
2. Каждое соединение имело бы ≤5 стримов — сервер легко справляется с 8
3. Все 5 стримов получили бы ответы
4. Таймаут `read=120s` тоже работал бы лучше: с меньшим числом стримов на соединении, когда сервер закончил обработку активных, сокет замолкает, и `anyio.fail_after(120)` корректно срабатывает

С 10 агентами и 4 воркерами: каждый воркер получает ~2-3 запроса → 1 соединение с ≤3 стримами → всё работает.

**Проблема:** сейчас **нет способа** настроить `max_concurrent_streams`. Нужно добавить.

## Что нужно сделать (варианты фикса)

### Вариант A: Config-опция для cap `max_concurrent_streams` (рекомендую)

Добавить поле в `HttpClientPoolConfig` или `HttpClientConfig`:
```python
max_concurrent_streams_per_connection: int | None = None
```

В `FixedHTTP2Connection._receive_remote_settings_change` (`h2_connection.py:200-203`):
```python
new_max_streams = min(
    max_concurrent_streams.new_value,
    self._h2_state.local_settings.max_concurrent_streams,
    self._max_streams_cap,  # НОВОЕ: config override
)
```

В `handle_async_request` (`h2_connection.py:298-309`): тоже применить cap при инициализации.

Это заставит пул открывать новые соединения после N стримов — именно то, что ты описал.

### Вариант B: Per-provider `http2: false`

Добавить `http2: bool` в `ProviderConfig`, передавать в `HttpClientFactory.get_client_for_provider()`. Для qwen-home выставить `http2: false` → HTTP/1.1 → каждое соединение = один запрос. Но теряем H2 для этого провайдера.

### Вариант C: Per-stream таймаут (сложнее, но правильнее)

Обернуть ожидание ответа для каждого стрима в `anyio.fail_after(read_timeout)` на уровне `FixedHTTP2Connection` или gateway. Это исправило бы баг №1 (таймаут не срабатывает для голодных стримов). Но сложнее реализовать корректно.

### Быстрый workaround прямо сейчас

В конфиге:
```yaml
http_client:
  http2: false  # глобально для всех провайдеров
```

Это отключит H2 вообще. Каждое соединение = один запрос, как в обычном HTTP. Но это влияет на ВСЕ провайдеры, не только qwen-home.
